# Q&A Notes

## 1. 召回 Recall@K 是怎么计算的？

Recall@K 的分母不是全量用户乘全量视频，而是当前 split 里真实发生的正反馈 `(user_id, video_id)` 对。  
我先定义正反馈，然后看召回 topK 候选里命中了多少真实正反馈：

`Recall@K = hit positive pairs in topK / all positive pairs in split`

这里的重点是评估候选池覆盖，而不是最终排序。

## 2. 为什么要定义 is_positive？

因为短视频场景里“正反馈”不应该只等于点赞。  
很多用户并不会点赞，但如果一个视频被长时间观看或者完整看完，这也是有价值的正反馈。

所以我定义：

- `like`
- `finish`
- `long_watch`

然后做：

- `is_positive = like OR finish OR long_watch`

这个定义更贴近短视频推荐，而不是把任务硬做成传统点击预测。

## 3. 为什么未观察到正反馈不等于负反馈？

因为这是隐式反馈场景。

如果某个候选在当前 split 没有观察到正反馈，只能说“没有观察到正反馈”，不能直接等于“用户不喜欢”。  
原因包括：

- 曝光偏差
- 上下文时机不合适
- 用户没真正消费到内容
- 用户兴趣不稳定

所以训练里用到的 `label=0` 本质上是弱负样本或未观察正样本，不是显式 dislike。

## 4. 为什么粗排后 Recall@100 还能提升？

这里要分两个口径。

粗排不可能提升“全候选池上限召回”，因为它本质上是过滤。  
但如果拿“召回层原始 top100”和“粗排后的 top100”比较，粗排后的 top100 可以更好，因为它重新排序了召回候选，把更可能命中的内容提前了。

所以出现：

- 召回层 top100 的绝对 Recall 较低
- 粗排后的 top100 绝对 Recall 更高

这是正常的，说明粗排在 top100 预算下选得更好。

## 5. 为什么 Graph Embedding 加入后先拉低指标，后来又调权重？

因为 Graph Embedding 的候选量一开始太大，直接挤占了原来更稳定的 `itemcf / twotower / popular` 候选。

Graph Embedding 的作用更适合做补充召回，而不是主召回路。  
所以我后面把：

- `source_weights.graph_emb`
- `per_source_topk.graph_emb`

都调低，让它以更小配额进入 merge。这样它还能提供结构化补充召回，但不会主导候选池。

## 6. 为什么精排选 DIN 而不是 Transformer？

主要是工程复杂度和项目定位。

DIN 的优势：

- target-aware attention，适合建模“当前候选”和“历史行为”的匹配
- 比纯 MLP 更适合短视频序列兴趣建模
- 比 Transformer 更轻量、更稳定、更容易单机跑通

如果这是个人项目第一版，我更看重“完整跑通 + 易讲清楚 + 工程可控”，所以 DIN 是更合理的折中。

## 7. 为什么重排不用学习型模型？

因为重排层要解决的是最后一层的业务约束问题：

- 作者频控
- tag 多样性
- 新鲜度调节

这些约束用规则式 rerank 有几个优势：

- 可解释
- 可控
- 不需要额外的 slate 级监督数据
- 很容易做策略调参

学习型 reranker 当然是后续方向，但不是第一版最优先的投入点。

## 8. 这个项目离真实工业系统还有哪些差距？

主要差距有几个：

- 没有在线 serving、ANN 服务和特征服务
- 没有流式实时特征更新
- 没有曝光偏差校正和线上 A/B test
- 没有大规模分布式训练与检索系统
- 重排还是规则式，不是 slate-aware learning-to-rerank

但它已经完整体现了工业推荐系统的核心分层思想和训练数据构造逻辑。

## 9. 如果继续优化，下一步会做什么？

我会优先做这几件事：

1. 提升精排层
   - 尝试 Transformer ranker 或更强的 DIN 变体
2. 提升召回层
   - 对 twotower 和 graph_emb 做更系统的 ablation 和调参
3. 做 bias handling
   - 引入 IPS / DR / unbiased evaluation
4. 升级重排层
   - 从规则式扩展到 learning-to-rerank

如果从“面试和项目价值最大化”的角度看，我会先继续做精排和 debias。

## 10. 这个项目最值得讲的亮点是什么？

我觉得最值得讲的不是某一个模型，而是这几点组合起来：

- 它是完整的四层推荐系统，不是单模型 demo
- 召回层做了多路融合，并且包含 Graph Embedding
- 粗排、精排、重排各自边界清晰
- 精排不是简单 MLP，而是 DIN-style 多任务模型
- 中间候选结果落盘，严格按 cascade 构造训练样本

这能说明我不是只会训模型，而是理解推荐系统为什么要这样设计。
